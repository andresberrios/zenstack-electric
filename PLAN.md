## Access Control Semantics (from ZenStack docs)

The correct evaluation order is:
1. If **any** `@@deny` rule matches â†’ **denied**
2. If **any** `@@allow` rule matches â†’ **allowed**
3. Otherwise â†’ **denied** (secure by default)

Expressed as SQL: `(NOT any_deny) AND (any_allow)`.

Operation types: `read`, `create`, `update`, `delete`, `post-update`, `all` (= create+read+update+delete).
For Electric shapes (read-only), only rules with operation `read` or `all` are relevant.
Comma-separated operations are valid: `@@allow('create,read', condition)`.

### Task 1: Fix NULL comparison â€” `IS NULL` / `IS NOT NULL`

**Bug:** `src/compile.ts:127-128` returns raw `'NULL'` for null expressions. When used in
`compileComparison` (line 231-258), this produces `"field" = NULL` which is always false in SQL.
Standard SQL requires `IS NULL` / `IS NOT NULL`. See PostgreSQL docs on NULL comparison:
https://www.postgresql.org/docs/current/functions-comparison.html

**Fix location:** `compileComparison` in `src/compile.ts` (around line 231). Before building
`${lSide.sql} ${sqlOp} ${rSide.sql}`, check if either side compiled to `'NULL'` and emit
`"field" IS NULL` or `"field" IS NOT NULL` instead. The `case 'null'` at line 127 can stay
as-is â€” the special handling belongs in the comparison.

**Existing test to update:** `test/index.test.ts` line 461-472 (`nULL handling` describe block)
currently expects `"deletedAt" = NULL` â€” update to expect `"deletedAt" IS NULL`.

- [x] In `compileComparison`, detect when either side is `'NULL'` and emit `IS NULL` / `IS NOT NULL`
- [x] Add unit test: `field == null` â†’ `"deletedAt" IS NULL`
- [x] Add unit test: `field != null` â†’ `"deletedAt" IS NOT NULL`
- [x] Add unit test: `null == field` (reversed) â†’ `"deletedAt" IS NULL`
- [x] Update existing NULL test (line 467) to expect `IS NULL`

**Files:** `src/compile.ts`, `test/index.test.ts`

### Task 2: Implement `@@deny` rule support

**Bug:** `src/compile.ts:58-62` only extracts `@@allow` attributes. `@@deny` rules are silently
ignored. Per ZenStack docs (https://zenstack.dev/docs/orm/access-control/write-policies),
deny rules take absolute precedence: any matching deny blocks access regardless of allows.

**Correct SQL semantics:** `(NOT (deny1 OR deny2 ...)) AND (allow1 OR allow2 ...)`.
If there are no deny rules, omit the NOT clause. If there are no allow rules, deny all.

**Fix location:** `compileModelFilter` in `src/compile.ts` (line 50-90). Add a second filter
pass for `@@deny` attributes. Compile deny conditions the same way as allows, then combine
the results. Short-circuit: if any deny is unconditional (`true` â†’ compiles to `null`),
the whole filter is `{ where: 'false', params: [] }`.

**Test helpers:** Use the existing `allow()` helper at `test/index.test.ts:37-45` as a template
to create a `deny()` helper that produces `{ name: '@@deny', args: [...] }`.

- [x] Add `deny()` test helper function in `test/index.test.ts` (mirror `allow()`)
- [x] In `compileModelFilter`, extract `@@deny` attributes alongside `@@allow`
- [x] Compile deny conditions and combine as: `(NOT (deny1 OR deny2 ...)) AND (allow1 OR allow2 ...)`
- [x] If any deny rule is unconditional (`true`), short-circuit to `{ where: 'false', params: [] }`
- [x] If deny has `false` condition, skip it (deny nothing = no effect)
- [x] Add unit test: `@@allow('all', true) + @@deny('all', status == 'DELETED')` â†’ `NOT ("status" = $1)`
- [x] Add unit test: `@@deny('all', true)` â†’ `{ where: 'false' }` regardless of allow rules
- [x] Add unit test: `@@deny('all', false) + @@allow('all', true)` â†’ `null` (deny false = no-op)
- [x] Add unit test: multiple deny rules combined with OR inside the NOT
- [x] Add unit test: deny + multiple allow rules compose correctly

**Files:** `src/compile.ts`, `test/index.test.ts`
